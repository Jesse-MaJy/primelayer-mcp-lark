# 测试中心 JSON 2.0 与 VChart Demo 卡片设计

## 背景

管理后台测试中心目前提供六个飞书卡片预设，但预设仍使用 JSON 1.0 的根级 `elements`、`lark_md`、`action` 和 `note` 结构。“AI 回答卡片”虽然携带 ECharts 参数，但该参数只是按钮回调值，飞书不会将其渲染成图表。这使测试中心无法验证生产回答卡已经采用的 JSON 2.0、原生表格和 VChart 能力。

## 目标

- 将六个测试卡片预设全部升级为飞书 JSON 2.0。
- 每个业务场景至少包含一个真正可渲染的 VChart 图表。
- 同时提供 Roche 单项目连续七天趋势和 Roche、Siemens、XDL 多项目横向对比数据。
- 保留测试中心现有的预设选择、JSON 编辑、格式化、单人发送和批量发送能力。
- 通过后端自动测试保证 Demo 卡片不再回退到旧版组件协议。

## 非目标

- 不在管理后台浏览器中模拟飞书卡片视觉渲染。
- 不引入 ECharts、图片图表服务或自定义网页组件。
- 不让 Demo 卡片替代正式 AI 回答卡渲染器。
- 不实现测试按钮对应的真实业务动作；按钮只验证 JSON 2.0 回调结构和飞书展示。

## 方案选择

采用后端 Demo 卡片目录服务，管理后台通过接口加载预设卡片。

相较于继续在 Vue 页面中拼装卡片 JSON，该方案可以用现有 Java/JUnit 工具验证协议结构，避免测试中心长期保留另一套不可测试的飞书协议。相较于把所有场景强行转换为 `AnswerPresentation`，独立 Demo 服务可以覆盖日报、风险提醒和综合组件测试需要的定制布局。

## 架构与数据流

新增一个仅服务于管理后台调试功能的 Demo 卡片目录组件。组件返回六个预设，每个预设包含：

- `key`
- `label`
- `description`
- `color`
- 完整的 JSON 2.0 `card`

新增只读管理接口返回上述目录：

`GET /api/admin/debug/feishu/card-presets`

管理后台加载测试中心时取得目录，使用服务端返回的元数据绘制选择器，并将选中预设的 `card` 格式化后放入现有 JSON 编辑框。发送流程保持不变：编辑后的 JSON 仍通过现有单发或批量调试接口发送。

若目录接口加载失败，页面展示错误消息并保留空状态，不静默回退到旧版 JSON 1.0 模板。

## JSON 2.0 组件规则

所有预设必须满足：

- 根节点声明 `schema: "2.0"`。
- 内容位于 `body.elements`。
- 正文使用顶层 `markdown` 组件。
- 明细使用原生 `table` 组件。
- 图表使用原生 `chart` 组件，`chart_spec` 只包含服务端定义的 VChart 配置。
- 图表类型限定为 `bar`、`line` 和 `pie`。
- 按钮回调通过 `behaviors` 声明，避免 JSON 1.0 顶层 `value`。
- 不出现根级 `elements`、`lark_md`、旧版 `action` 或 `note` 组件。

Demo 图表配置是固定、可信的服务端数据，不接受页面或模型提供的任意 JavaScript、HTML 或动态表达式。

## 六个预设

### AI 回答卡片

- 项目结论 Markdown。
- 项目 KPI 原生表格。
- Roche 七天计划与实际进度折线图。
- 任务状态占比饼图。
- 项目详情测试按钮。

### 基础测试卡片

- Markdown 标题、列表、加粗和链接示例。
- 小型数据表格。
- 柱状图、折线图、饼图各一张，用于一次性验证三类 VChart。
- JSON 2.0 回调按钮。

### 每日待办

- 今日任务摘要。
- 待办优先级占比饼图。
- 包含负责人、项目、优先级、截止时间和状态的待办表格。
- Demo 数据同时覆盖 Roche、Siemens 和 XDL。

### 施工日报

- Roche 七天计划/实际完成率折线图。
- 七天现场人员投入柱状图。
- 当日质量问题明细表格。
- 当日完成、异常风险和明日计划 Markdown。

### 风险提醒

- Roche、Siemens、XDL 的高/中/低风险数量对比柱状图。
- 风险类别占比饼图。
- 包含影响节点、责任人、截止时间和状态的风险明细表格。

### 周报摘要

- 三项目计划/实际完成率横向柱状图。
- 七天新增与关闭问题趋势折线图。
- 三项目周度汇总表格。
- 本周亮点、阻塞项和下周重点 Markdown。

## Demo 数据规模

- 单项目趋势固定为连续七天，每个系列七个点。
- 多项目比较固定包含 Roche、Siemens 和 XDL。
- 表格根据场景提供 5 至 12 行，足以验证行渲染、长文本和多状态数据。
- 每张卡最多三张图，避免测试消息过长并与正式回答卡的限制保持一致。
- 日期标签运行时生成，业务数字固定，保证测试结果可复现。

## 代码边界

- Demo 卡片目录服务只负责测试预设和可信 VChart spec。
- `DebugController`/`DebugService` 只暴露目录和现有发送能力，不承担卡片拼装细节。
- `TestCenterView.vue` 只负责选择、编辑和发送，不再维护飞书组件构造函数。
- 正式回答仍由 `FeishuAnswerCardRenderer` 生成，Demo 服务不得改变正式消息行为。

## 错误处理

- 目录加载失败：测试中心显示明确错误，卡片选择器保持不可发送的空状态。
- 返回目录为空或当前 key 不存在：选择首个有效预设；没有有效预设时不生成 JSON。
- 用户手工编辑出非法 JSON：沿用当前页面校验并阻止发送。
- 飞书拒绝卡片：沿用现有调试发送接口原样展示错误，方便定位具体组件问题。

## 测试与验收

后端先编写失败测试，再实现目录服务：

- 恰好返回六个唯一预设。
- 每个卡片声明 JSON 2.0，元素位于 `body`。
- 每个预设至少包含一个 `chart`，综合预设覆盖 `bar`、`line`、`pie`。
- 单项目趋势包含七个日期点，多项目比较包含三个项目。
- 预设中不存在根级 `elements`、`lark_md`、旧版 `action` 和 `note`。
- 表格、按钮和 VChart spec 使用预期 JSON 2.0 字段。

前端通过类型检查和生产构建验证接口类型、空状态、预设切换和 JSON 编辑逻辑。最后运行：

```bash
cd backend && mvn test
cd admin-web && npm run build
```

验收时从测试中心分别发送六个预设，确认新版飞书客户端可展示 Markdown、表格、柱状图、折线图、饼图和按钮。
